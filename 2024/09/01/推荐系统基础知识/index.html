¶推荐系统

¶基本架构、术语和意义

搜广推：搜索广告推荐

三种场景使用的技术路线大致相似

共同点：解决信息过载的问题 从海量信息中找出用户的目标信息

推荐系统一定要有一个内容池corpus

¶推荐系统核心指标

电商：购买率

抖音：用户数（DAU 日活）*平均时长

短视频平台4方博弈：用户 内容生产者（个人/机构）平台方 广告方

不同时期，不同目标，通过推荐系统，完成流量分配

¶新架构和技术指标

  1. 用户上传
  2. 内容审核：人工审核+机器审核

​ 重视召回率高而不是准确率 宁可错杀不可放过

  1. 内容理解（打标）
    • 内容分类
    • 标签抽取：提取关键词
    • 内容向量化
    • 标签的生命周期（新闻过时效后不再推流）
    • 初始流量预测
  2. 召回：快速全面从上千万数据中找到几百个
  3. 排序：准确从几百个里找到最合适的20个
  4. shuffle打散
    • 增强多样性
    • 插入合适广告
  5. 埋点：记录用户行为
    • 给用户曝光的item
    • 用户点击的item：点赞评论留言

评价指标：

  • 点击率：点击/曝光（不同用户的点击率平均）

  • 完播率：观看时长/视频时长（整体统计时长分布，根据长度分组）

  • 日活：新用户点击率和时长 老用户点击率和时长

指标总在变化：周末>平时 晚上>白天

推荐策略也应该发生变化（召回和排序）

比如白天多推新闻 晚上多推人生哲理

¶数据预处理

feed_info.csv

discription:描述中的每一个字对应一个id（映射）

feed_embeddings.csv：将视频转为向量化表示，可以进行不同视频相似度计算

train.csv

记录方式：从打开app后有很多埋点，会记录一切行为

数据预处理目的：生成每个用户对视频的喜好程度

userid -> video_score（根据csv中数据计算而来）

¶协同过滤 – 经典的推荐算法

是一种召回算法

协同：用很多用户的行为一起进行推荐

过滤：从海量视频->有限个视频

协同过滤：协同大家的反馈、评价和意见一起对海量的信息进行过滤，从中筛选出目标用户可能感兴趣的信息的推荐过程

¶itemCF

从item的角度进行过滤筛选：很多人都看了item1和item2 则其存在着某种程度的相似 比如同时看女生跳舞和俄乌局势（宅男视角）

当大量用户同时观看item1和item2时，可以将item2推荐给观看了item1的用户，即使内容上可能毫无关联，但在受众群体上可能有高相似度

¶userCF

两个用户的观看视频中存在大量相似，则其中一个用户看了一个新视频的情况下，可以将其推荐给另一个用户

¶协同过滤的问题和解决方法

  • 相似的不对称

item1的相似视频有item2 但item2的相似视频没有item1

userCF同样存在这个问题

协同过滤是利用过去的点击行为推荐未来

问题：热门视频被推荐的次数越来越多，冷门视频越来越冷门

不仅协同过滤存在这个问题，召回方法都存在这个问题

  • 冷启动

对于新用户新item来说协同过滤是失效的

新用户：推荐热门、同地域、同年龄、同自然属性、同社会属性

新item：社交网络推荐、同地域、内容推荐（feed_embeddings计算相似度）

内容类app：item冷启动

电商类app：用户冷启动

抖音 vs 快手的区别：

  • 抖音：热门推荐，流量集中，易出现流量聚集在热门视频情况
  • 快手：同地域社交推荐

推荐系统不独立存在，和上下游息息相关

¶随机游走算法

也是召回算法

前面和itemCF一样，获取商品之间的相似度

商品之间互相关联，形成了一个网（item为点 关联为线）

网中item互相到达有很多种序列

将序列进行word2vec处理变成向量，即随机游走向量

随机游走向量 vs 内容向量：

  • 内容向量相似完全由item内容决定，说明内容相似
  • 随机游走向量相似是行为造成的，说明行为相似，可能跟内容毫无关系

随机游走 vs itemCF：

随机游走泛化能力更强，任何两个向量都可以计算相似度，可以推荐的范围更广。比如：item1和item2有相似度，item2和item3有相似度，通过itemCF无法通过item1推荐item3，但随机游走可以实现这种间接推荐。有利有弊，范围更广但是可能会推荐错。

¶DSSM召回模型（双塔模型）

item1通过model1生成向量1

Item2通过model2生成向量2

向量可以离线生成

model1和model2是否一样取决于item是否同质，同质化内容可以用同一模型，但不同内容比如图像和文字就要分别用不同模型

向量1和向量2仅仅是用来独立计算相似度的

DSSM如何训练：

每个item采若干个正样本（比如点击过的）和若干个负样本

S0 = <v,v0> S1 = <v, v1>…（s表示分数）

训练目标 正样本s越高越好，负样本s越低越好

可以进行softmax处理，当成多分类问题

本质：通过计算user和item的相似度，对user进行高相似item的推荐

¶多路召回的merge策略

使用不同的召回策略得到的结果可能不同

比如，对于user1来说，item1被5种策略召回，即5路召回，item2被3种策略召回，即3路召回。

¶推荐系统的AB测试

AB测试用于测试新策略是否优秀

baseline：有部分原有召回策略

s：新召回策略

加入s1的效果明显优于A，则将s1加入baseline

AB测试的时间一般在3-5天

问题：新策略中大量增多低俗视频，可能会在短期产生好效果，但长期不利于平台

反向AB测试：baseline中策略太多，需要下线一些策略

同理测试去除策略与baseline效果进行比较

分组的个数： N总量 n组 N/n个用户

n太大，每个组用户少，统计结果随机性大

n太小，一次能够进行实验的策略少，开发效率低

¶分层AB测试

总人数不够多的情况下 如何分组能保证效果

随机方法1将10w用户打散为5组 得到层1

随机方法2将每组中的2w用户打散为5组 重新混合为5组2w 得到层2

因此得到了更多组，可以分别进行实验

但不可以隔层进行对比

¶召回和倒排索引

召回：

  1. 快速从海量数据中提取相关的item
  2. 倒排索引形式

正排索引

Doc1 A B C

Doc2 B C D

倒排索引（类似于字典）

A Doc1

B Doc1 Doc2

C Doc1 Doc2

D Doc2

倒排的优点：用户搜索B，会迅速返回doc1和doc2给用户，不需要遍历文章

优化：将文章和词的关联得分也记录

A Doc1 0.9

B Doc1 0.8 Doc2 0.7

C Doc1 Doc2

D Doc2

¶召回和排序的分工和区别

召回：

  1. 快速从海量数据中提取相关item

  2. 倒排索引形式 item1 -> item2, item5, item6 （可以提前离线处理）

  3. 倒排索引的产生

    a 行为类 协同过滤 随机游走

    优点：不需要了解内容 只通过用户行为

    缺点：易推荐热门 形成信息茧房

    b 内容类 通过内容信息进行推荐：同关键词 同作者 相似文章向量

  4. 召回模式

    统计类：协同过滤 更保守，至少共同出现过一次才推荐

    向量类：行为和内容都可以 更激进，任意两个item都有可能被推荐

排序：召回得到的海量数据排序，得到前若干个进行推荐

典型的排序模型：LR逻辑回归
