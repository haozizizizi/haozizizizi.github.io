¶多目标建模

推荐系统中的多目标优化是为了同时满足多个目标的需求而进行的优化任务。在实际应用中，推荐系统通常不仅仅关注一个目标（如点击率、购买率），而是需要综合考虑多个目标，以提升用户体验和系统的综合效益。

¶目标

CTR（Click-Through Rate，点击率）：

  • 点击率是推荐系统中衡量用户对推荐结果感兴趣程度的关键指标之一，表示用户点击某个推荐内容的比例。公式为：CTR = 点击次数 / 展示次数
  • CTR 越高，表示推荐内容越能吸引用户的注意力和兴趣。

CVR（Conversion Rate，转化率）：

  • 转化率是衡量用户从点击推荐到完成某一目标动作（如购买、下载、注册等）的比例。它通常是推荐系统中衡量推荐效果是否产生实际商业价值的重要指标。公式为：CVR = 转化次数 / 点击次数
  • CVR 越高，表示推荐内容不仅能吸引用户，还能促使他们采取进一步的行动。

CTCVR（Click-Through Conversion Rate，点击转化率）：

  • 推荐系统或广告系统中一个重要的指标，用于衡量从用户点击推荐内容到最终完成转化（如购买、注册、下载等）的整体转化效果。

  • CTCVR 是结合了点击率（CTR）和转化率（CVR）的综合指标，具体来说，它代表了从展示到点击，再到实际转化整个链条的转化概率。公式为：CTCVR = 转化次数 / 展示次数

¶Q. 为什么不为单个目标建模

  1. 首先，单独为每个目标建模会浪费大量资源。大型的推荐模型需要耗费很多内存、计算力和时间，而如果每个目标都独立建模，成本会大幅增加，效率低下。

  2. 其次，用户的转化过程是一个链条，比如用户先点击、再加入购物车、最后购买。链条中越靠后的环节价值越大，但可用的正样本越少。单独训练会导致embedding稀疏影响效果，增大误差。联合训练可以帮助知识在整个链条上进行转移，前期数据的影响能够更好地传递到后期。

¶Q.为什么不直接建模高级目标？

以电商场景为例，为什么不直接建模最关注的CTCVR，而是同时建模CTR、CVR、CTCVR

  1. 首先，用户最终没有购买，并不代表用户就一定不喜欢推荐结果，也很有可能是因为商品价格超出了这名用户的消费能力。如果只以提高CTCVR为唯一目标，APP推荐给这名用户的就都是在他消费能力之内的中低端商品。可能会暂时提高销售额，但是会带来两方面的危害：不仅容易造成用户的审美疲劳，对用户的长期留存不利。并且，也失去给用户"种草"的机会。如果用户某天想购买更加昂贵的产品，APP却推荐不出来高端商品，也就浪费了一次提高销售额的机会。因此，尽管CTCVR对提高当前、短期的销售额很重要，但不能作为唯一目标。优化CTR、CVR环节有利于提高用户粘性和长期销售额。
  2. 其次，无论是否转化，用户的每次点击都有价值。在广告定价时，会针对CTR、CVR、CTCVR设计不同的计费模式。所以，三个概率都要建模，并且都要预估得尽量准确。
  3. 最后，只建模CTCVR，可用数据少，导致训练不充分，误差比较大。因此需要联合其他目标，使前期数据传递到后期

多目标建模目前业内有两种模式，一种叫Shared-Bottom模式，另一种叫MOE，MOE又包含MMOE和OMOE两种。

¶Shared-Bottom

思路：多个目标底层共用一套共享layer，在这之上基于不同的目标构建不同的Tower。这样的好处就是底层的layer复用，减少计算量，同时也可以防止过拟合的情况出现。

优点：

  • 知识迁移：通过联合训练多个任务，实现了“知识迁移”。例如，任务A的数据量充足，而任务B的数据量较少。在联合训练时，任务A的丰富数据可以帮助任务B，增强任务B的训练效果，使得任务B在较好的状态下继续训练，从而事半功倍。

缺点：

  • 过度理想化

    虽然这种理论有一定的逻辑基础，但实践中不同任务之间的关系并非总是相辅相成。不同任务在共享底层参数时，可能会出现负面效果。这导致了两种现象：

    1. 负迁移现象（Negative Transfer）：联合训练任务A和B时，任务A的训练效果比单独训练任务A更差，任务B同样表现不佳。
    2. 跷跷板现象（Seesaw Effect）：联合训练的结果使得任务A的效果比单独训练A更好，但任务B的效果比单独训练B差，甚至可能反转。这意味着优化一个任务时，可能会以牺牲另一个任务为代价。
¶MOE

Shared-Bottom是一种Hard parameter sharing，会导致不相关任务联合学习效果不佳，为了解决这个问题，Google提出了Soft parameter sharing，MOE是其中的一种实现。

MOE（Mixture of Experts）模型用于解决不同任务之间的冲突问题，并提升模型的灵活性和性能。MOE模型通过引入“专家网络”（Experts）和“门控网络”（Gating Network），让不同任务可以根据输入数据动态选择最适合的专家进行学习和预测。

MOE结构的算法底层是由多个expert组成，这些expert网络是彼此独立的，最终的输出结果其实是目标在多个expert上的概率分布结果。MOE可以通过一个Gate network去控制，Gate network也叫门控网络，用来控制在不同应用场景下每个expert的参与权重。

¶MMOE

MMOE(Multi-gate Mixture-of-Experts)是在MOE的基础上，使用了多个门控网络， k个任务就对应k个门控网络，这样不同的专家可以专注于不同的任务，同时所有专家为所有任务共享，共享程度由门控网络根据当前样本动态调节。

Q.如果出现极化现象，导致门控网络softmax输出接近1，其余接近0如何解决？

可以对softmax输出加上dropout，强制要求每个专家参与到每个任务。

MMOE只能减轻多目标中的负迁移现象，但跷跷板现象还是广泛存在的，并且MMOE还有两个缺点：

  1. MMOE中所有的Expert是被所有任务共享的，可能无法捕捉任务之间更复杂的关系，从而给部分任务带来一定噪声
  2. 不同的Expert之间没有交互，联合优化效果有折扣
¶PLE

PLE 通过对不同任务分别保留独立的专家（Experts），同时共享部分专家，实现任务间的差异化建模和信息共享。这种结构既能保证每个任务的个性化处理，也能在必要时实现信息共享，避免任务间过度干扰和负迁移。

针对不同Expert之间没有交互的问题，PLE提出了多级提取网络，任务k的独立专家模块融合了上一层
网络中任务k的独立专家模块和共享专家模块，而共享专家模块则融合了上一层所有的专家模块。

¶ESMM

ESMM（Entire Space Multi-Task Model），即全空间多任务模型，该模型主要用于解决CVR（转化率）预测中的两个常见问题：样本选择偏差（SSB） 和 数据稀疏性（DS）。

ESMM模型的两个核心问题：

  1. Sample Selection Bias（SSB，样本选择偏差）：
    • CVR 是基于点击后发生的事件，传统的CVR模型通常只使用点击后的数据进行训练。然而，点击样本通常是转化样本的一部分，这导致训练好的CVR模型难以在未点击样本上有效预测。换句话说，CVR模型只对点击样本进行了预测，无法对整个样本空间进行泛化，这导致模型预测效果受限。
  2. Data Sparsity（DS，数据稀疏性）：
    • CVR数据的训练样本通常要小于CTR的训练样本。这是因为点击后转化的样本相较于展示样本来说更少，导致转化率训练中的数据非常稀疏。数据稀疏性问题使得CVR模型难以有效学习和预测。

考虑到曝光 -> 点击 -> 转化这个行为链路是具有依赖关系的，曝光 -> 点击建模的是CTR，曝光 -> 点击 -> 转化建模的是CTCVR，点击 -> 转化建模的是CVR，这三者存在数学上的依赖关系

并且CTR和CTCVR的样本空间都是曝光样本，因此这两个任务是可以使用全部样本的，因此考虑通过学习这两个任务，隐式地学习CVR。

ESMM构建两个塔(底层共享参数，可以实现知识迁移，解决CVR样本稀疏的问题）分别预估pCTR和pCVR，然后将pCTR和pCVR相乘得到pCTCVR，然后用pCTR和pCTCVR去计算损失后
